I started this project after hearing about [OpenClaw](https://github.com/Upsonic/OpenClaw)—an open-source, self-hosted AI agent that uses Telegram not just for chatting, but as an interface to execute real-world tasks. It made me realize that a companion bot could be more than just a prompt-responder. The initial idea was simple: build a Python agent on Telegram using Claude. But as I explored the APIs, it became clear that a raw chat window is a weak point. An LLM on its own is just a conversationalist; it doesn't have a "life" outside the text box.

To fix this, I built a Telegram Mini App that lives alongside the bot. The Mini App provides the features that keep users coming back: journals, motivation quotes, and song recommendations. But there's a catch: to keep the Mini App "alive," you have to talk to the bot.

---

## The Engagement Loop

The companion is deployed on Railway and uses a webhook-based FastAPI server. The core philosophy is split into three panels in the Mini App, representing a person's timeline:

1.  **Past**: Journals written by the bot about your conversations.
2.  **Present**: Daily motivation quotes and "unhinged" things you've said. 
3.  **Future**: Still undecided, but currently tracking notes and tasks.

The bot is the "brain," but the Mini App is the "memory." If you don't talk to the bot, the journals don't update, the daily quote stays the same, and the relationship stalls.

## Architecture: Co-hosting Webhooks and UI

One of the first technical hurdles was running the Telegram bot, a REST API, and a frontend from a single process. 

I used FastAPI's lifespan pattern to manage the bot's lifecycle alongside the web server. This allows the same Railway service to handle incoming Telegram webhooks while serving the Mini App's static assets.

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup: Start the bot
    await bot.start()

    if settings.WEBHOOK_URL:
        await bot.application.bot.set_webhook(
            url=f"{settings.WEBHOOK_URL}/webhook/{settings.TELEGRAM_BOT_TOKEN}"
        )

    yield  # App is running

    # Shutdown
    await bot.stop()

app = FastAPI(title="Companion API", lifespan=lifespan)
```

For the Mini App UI, I optimized for the embedded WebView's latency by fetching everything in a single dashboard endpoint. One request, all tabs.

```python
@app.get("/api/dashboard/{telegram_id}")
```

## Building the "Past": Automated Journaling

The "Past" tab is powered by a tiered memory system. Every 30 messages, the system triggers a summarization task. It doesn't just record facts; it writes a "journal entry" from the bot's perspective. 

This is where the companion feels different from a generic assistant. It's not just a log; it's a reflection.

### Handling Fragmented Input (Debouncing)

Users send rapid-fire messages. If the bot responds to every "hey" and "quick question," it costs a fortune and feels disjointed. I implemented a simple 0.5s debounce timer:

```python
async def handle_text_message(self, update, context):
    chat_id = update.effective_chat.id
    message_text = update.message.text

    if chat_id not in self._message_buffers:
        self._message_buffers[chat_id] = []
    self._message_buffers[chat_id].append(message_text)

    if chat_id in self._debounce_tasks:
        self._debounce_tasks[chat_id].cancel()

    self._debounce_tasks[chat_id] = asyncio.create_task(
        self._process_buffered_messages(chat_id)
    )
```

This ensures we process a complete thought, making the journals much more coherent.

### Memory Creation: Reflection, Not Just Summary

The system generates two types of summaries: a factual compact summary and an emotionally-aware conversation memory. The latter forms the basis of the journal entries.

```python
# COMPACT_PROMPT — factual, third-person record keeping
"""Create a factual, unbiased summary of this recent exchange.
Include timestamps, key details, and any commitments made."""

# MEMORY_PROMPT — personal, empathetic memory formation
"""Write as if remembering someone you care about.
What would you want to remember about them from this exchange?"""
```

The trigger for these memory creations is database-based, ensuring persistence across server restarts.

## The "Present": Daily Insights and Identity

The middle tab is for engagement. It generates motivation quotes based on conversation history and tracks "iconic" things the user has said. Recently, I added a Spotify integration for daily song recommendations.

These features make the Mini App something a user wants to check regularly. But the bot still has limits. It wasn't designed for heavy tool use or complex MCP calls.
