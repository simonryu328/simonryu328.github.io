Most AI chatbot tutorials end at the same place: call the API, return the response. What they skip is everything that makes the difference between a toy and something you'd actually use — message timing, cost management, memory that survives restarts, and making it feel like someone's actually there.

This is a writeup on **Aki**, a personal AI companion I built on Telegram. The codebase is ~5,000 lines of Python across FastAPI, python-telegram-bot, Claude, PostgreSQL, and LiteLLM. I'll go through the engineering decisions that mattered — co-hosting a webhook and API server, why debouncing is non-negotiable, how memory gets tiered so costs stay manageable, and what I actually learned once real people started using it.

> The source code is on GitHub, found [here](https://github.com/simonryu328/aki).

## Co-hosting a Webhook and an API

The first design question: how do you run a Telegram bot, a REST API, and a frontend all from one process?

Most Telegram bot tutorials use polling — the bot connects to Telegram and asks "any new messages?" in a loop. Fine for development. In production you want webhooks: Telegram pushes to your server, which is more efficient and scales better.

Aki's FastAPI server also needs to serve the Mini App frontend (static HTML/JS/CSS) and expose REST endpoints for it to call. The fix: FastAPI's lifespan pattern starts the bot alongside the API server.

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup
    await bot.start()

    if settings.WEBHOOK_URL:
        await bot.application.bot.set_webhook(
            url=f"{settings.WEBHOOK_URL}/webhook/{settings.TELEGRAM_BOT_TOKEN}"
        )

    yield  # App is running

    # Shutdown
    await bot.stop()

app = FastAPI(title="AI Companion API", lifespan=lifespan)
```

`WEBHOOK_URL` acts as a toggle — set it and the bot uses webhooks; leave it empty and it falls back to polling for local development.

The Mini App loads everything from a single endpoint:

```python
@app.get("/api/dashboard/{telegram_id}")
async def get_dashboard(telegram_id: int):
    user = await memory_manager.get_or_create_user(telegram_id=telegram_id)

    memories   = await memory_manager.get_diary_entries(user.id, limit=50)
    daily_msg  = await _get_daily_message_data(telegram_id, user)
    soundtrack = await _get_daily_soundtrack_data(telegram_id, user)
    insights   = await _get_personalized_insights_data(telegram_id, user)
    horizons   = await memory_manager.get_future_entries(user_id=user.id)

    return DashboardResponse(
        profile=UserProfileResponse(...),
        memories=memories,
        daily_message=daily_msg,
        soundtrack=soundtrack,
        insights=insights,
        horizons=horizons,
    )
```

One request, everything the UI needs. Telegram Mini Apps run inside an embedded WebView with worse network latency than a native browser — minimizing round-trips matters.

## Telegram Bot API — The Parts Nobody Talks About

### Message Debouncing

Real people don't send one long paragraph. They send 3-5 rapid-fire messages:

  <p style={{fontSize: "0.85rem", color: "#888", marginTop: "0.5rem"}}><em>Aki's own description of what it is — split naturally across multiple messages, the way people actually text</em></p>
</div>

If you process each message independently, you get 3-5 separate LLM calls, each missing the context of the others. The response is fragmented and costs 3-5x more than it should.

The fix is debouncing — buffer incoming messages and wait for a pause before processing them as one:

```python
async def handle_text_message(self, update, context):
    chat_id = update.effective_chat.id
    message_text = update.message.text

    if chat_id not in self._message_buffers:
        self._message_buffers[chat_id] = []
    self._message_buffers[chat_id].append(message_text)

    if chat_id in self._debounce_tasks:
        self._debounce_tasks[chat_id].cancel()

    self._debounce_tasks[chat_id] = asyncio.create_task(
        self._process_buffered_messages(chat_id)
    )
```

After 0.5 seconds of silence, all buffered messages get joined with `\n` and processed as one. The delay is configurable — too low and you split multi-message thoughts; too high and it feels slow.

### Rate Limiting

With LLM calls costing real money, rate limiting is not optional. Aki uses a sliding-window rate limiter — no Redis, just an in-memory dict:

```python
class RateLimiter:
    def __init__(self, max_messages: int, window_seconds: int):
        self.max_messages = max_messages
        self.window_seconds = window_seconds
        self._user_windows: Dict[int, List[float]] = {}

    def check_rate_limit(self, user_id: int) -> tuple[bool, int]:
        now = time.time()
        cutoff = now - self.window_seconds

        timestamps = self._user_windows.setdefault(user_id, [])
        timestamps[:] = [t for t in timestamps if t > cutoff]

        if len(timestamps) >= self.max_messages:
            return False, 0

        timestamps.append(now)
        return True, self.max_messages - len(timestamps)
```

Default: 10 messages per 60-second window. Rate limiting is checked *after* debouncing, so a burst of 5 rapid messages counts as 1 toward the limit.

### Proactive Reach-Outs

Most chatbots just wait for input. Aki initiates conversations. An APScheduler cron job periodically scans for users who haven't messaged in a while. When it finds one, it generates a reach-out:

The reach-out uses the same tiered context (memories + recent conversations) as regular responses, but with a specialized prompt that tells the LLM to pick up on *unfinished threads* — things the user mentioned but never resolved. Users can configure reach-out behavior per-user: enable/disable, minimum silence hours, maximum silence days — all via Telegram commands.

## The Memory System

The biggest limitation of most AI chatbots is amnesia. Every conversation starts from zero. Aki handles this with a tiered memory architecture:

```
System Prompt Context
│
├── Recent Exchanges (4 entries)
│   └── Memory entries with emotional context
│
├── Conversation Memories (diary)
│   └── Personal diary entries, triggered by milestones
│
└── Current Conversation (20 msgs)
    └── Raw messages from the DB (most recent)
```

### Layer 1: Raw Conversations

Every message is stored in the `conversations` table with a timestamp. Ground truth.

### Layer 2: Compact Summaries and Conversation Memories

Every 30 messages (configurable via `COMPACT_INTERVAL`), two things happen in parallel:

1. A **compact summary** — factual, timestamped, condensing the exchange
2. A **conversation memory** — an emotionally-aware reflection on what was significant

Different prompts, different purposes:

```python
# COMPACT_PROMPT — factual, third-person record keeping
"""Create a factual, unbiased summary of this recent exchange.
Include timestamps, key details, and any commitments made."""

# MEMORY_PROMPT — personal, empathetic memory formation
"""Write as if remembering someone you care about.
What would you want to remember about them from this exchange?"""
```

The trigger is database-based, not in-memory. The agent queries message count since the last diary entry — this means memory creation survives server restarts.

### Context Assembly: The Glue Overlap

When building the LLM's context, the agent uses a "glue overlap" strategy:

```python
async def _build_conversation_context(self, user_id, full_history, user):
    # 1. Get the 4 most recent memory entries
    memory_entries = await self.memory.get_diary_entries(
        user_id, limit=settings.MEMORY_ENTRY_LIMIT
    )

    # 2. Find where the latest memory ends
    # 3. Show raw messages AFTER that point + 3 overlap messages
    # This creates continuity between summarized and raw context
```

The 3 overlap messages prevent a gap between where the memory summary ends and where the raw conversation picks up. Without this, the LLM loses the conversational thread.

### Prompt Caching

Anthropic's prompt caching can save up to 90% on input costs — but only if you structure prompts correctly. The system prompt splits into two blocks:

```python
SYSTEM_STATIC = """[persona, response format, rules]
{persona}
...response format instructions...
"""  # ← Identical across calls → cacheable

SYSTEM_DYNAMIC = """
{recent_exchanges}
{conversation_history}
Current time: {current_time}
"""  # ← Changes every call → not cached
```

The LLM client applies `cache_control` breakpoints to the static block. With the Anthropic beta header:

```python
if has_cache_control and "claude" in model.lower():
    kwargs["extra_headers"] = {
        "anthropic-beta": "prompt-caching-2024-07-31"
    }
```

## Cost Engineering

An always-on AI companion is expensive if you're not paying attention. Every message, memory creation, daily message, reach-out — they all cost tokens. Here's how I kept it from getting out of hand.

### LiteLLM as the Abstraction Layer

Aki uses [LiteLLM](https://github.com/BerriAI/litellm) to abstract away provider differences. Swap models by changing a string:

```python
class LLMClient:
    @retry(stop=stop_after_attempt(3), wait=wait_exponential(...))
    async def chat_with_usage(self, model, messages, **kwargs):
        response = await litellm.acompletion(
            model=model,           # "claude-haiku-4-5-20251001"
            messages=messages,     # or "gpt-4o-mini"
            temperature=temperature,
            max_tokens=max_tokens,
        )

        return LLMResponse(
            content=response.choices[0].message.content,
            input_tokens=usage.prompt_tokens,
            output_tokens=usage.completion_tokens,
            cache_read_tokens=getattr(usage, "cache_read_input_tokens", 0),
            cache_creation_tokens=getattr(usage, "cache_creation_input_tokens", 0),
        )
```

The gotchas: cache token reporting differs between providers, and LiteLLM sometimes prepends `anthropic/` or `openai/` to model names. You need `getattr` with defaults everywhere, and your pricing lookup needs to handle both name formats.

### Five Model Slots

Not all tasks need the same model:

```python
MODEL_CONVERSATION  = "claude-haiku-4-5-20251001"  # Main chat
MODEL_PROACTIVE     = "claude-haiku-4-5-20251001"  # Reach-outs
MODEL_MEMORY        = "claude-haiku-4-5-20251001"  # Summaries
MODEL_DAILY_MESSAGE = "claude-haiku-4-5-20251001"  # Daily quotes
MODEL_INSIGHTS      = "claude-haiku-4-5-20251001"  # Personalized insights
```

During development I used Sonnet for everything — great quality, terrifying bill. In production, Haiku handles most tasks at ~10x lower cost with acceptable quality. The conversation model is usually the only slot where upgrading makes a noticeable difference.

### The Pricing Module

Every LLM call records token usage to the `token_usage` table with a `call_type` label. The pricing module calculates estimated costs, including Anthropic's cache multipliers:

```python
def calculate_cost(model, input_tokens, output_tokens,
                   cache_read_tokens=0, cache_creation_tokens=0):
    pricing = get_model_pricing(model)

    cost_input  = (input_tokens / 1_000_000) * pricing["input"]
    cost_output = (output_tokens / 1_000_000) * pricing["output"]

    cost_cache_write = (cache_creation_tokens / 1_000_000) * (pricing["input"] * 1.25)
    cost_cache_read  = (cache_read_tokens / 1_000_000) * (pricing["input"] * 0.1)

    return cost_input + cost_output + cost_cache_write + cost_cache_read
```

### Per-User Token Budgets

There's a `USER_DAILY_TOKEN_BUDGET` setting. When set, the orchestrator checks daily token consumption before every LLM call. One active user won't blow through your monthly budget in an evening.

## What Actually Mattered

A few things I'd tell myself at the start:

**Debouncing is non-negotiable.** Without it, rapid-fire messages produce fragmented, expensive responses. 0.5 seconds is the right window — long enough to catch multi-message thoughts, short enough to feel immediate.

**Build cost visibility before inviting anyone.** The Streamlit dashboard was one of the first things I built. By the time beta testers joined, I could see where every token was going.

**Database-based triggers, not in-memory counters.** Memory creation is triggered by querying message count in the database, not by a counter in RAM. The bot can restart or crash without losing track of when to create the next memory entry.

**Telegram Mini Apps are underrated.** Embedding a full Vite/TypeScript app inside Telegram via `MenuButtonWebApp` gives you a rich UI without asking users to download anything. The unified dashboard endpoint keeps it snappy.

**Cheap models for background tasks, better models for conversation.** Haiku for memory generation, reach-outs, and daily messages. Sonnet when it actually matters. The cost difference is significant; the quality difference for background tasks is marginal.

---

Aki started as a curiosity — what would it take to make an AI that actually knows you? The interesting problems turned out to have nothing to do with prompts. They were infrastructure: how you manage state across restarts, how you control costs before they spiral, how you handle the timing quirks of a platform built for humans texting each other.

If you're building something similar: get the boring parts right first. Your prompts will change constantly. The infrastructure has to work from day one.
